## `get_token` — 10 variants

The tokenizer function (`sqlite3GetToken`) is the most frequently changing fragment.

### v1: 3.12 - 3.24 (18 versions) — baseline

The original tokenizer with the aiClass character-class dispatch.

### v2: 3.25 - 3.32 (13 versions) — NUL handling

Added `case CC_NUL:` returning `TK_ILLEGAL` with length 0. Previously NUL bytes fell through to `default:` which returned length 1, causing the tokenizer to skip over embedded NULs instead of stopping. Bug fix.

### v3: 3.33 - 3.34 (3 versions) — fallthrough annotations

Added `/* no break */ deliberate_fall_through` annotations at two switch case fallthrough points (CC_DOT→CC_DIGIT and CC_X→CC_ID). Compiler warning suppression for `-Wimplicit-fallthrough`. No behavioral change.

### v4: 3.35 (2 versions) — CC_KYWD0 keyword optimization

`case CC_KYWD:` renamed to `case CC_KYWD0:`. Added a separate `case CC_KYWD:` that falls through directly to `CC_ID`, bypassing the keyword hash lookup. This is the tokenizer half of the keyword optimization: only characters that can start an SQL keyword (`CC_KYWD0`) enter the keyword hash path; subsequent keyword characters and non-keyword-starting letters skip it.

### v5: 3.36 - 3.37 (3 versions) — UTF-8 BOM handling

Added `case CC_BOM:` detecting the 3-byte UTF-8 BOM sequence (0xEF 0xBB 0xBF) and returning `TK_SPACE`. Gracefully skips byte order marks at the start of SQL input instead of treating them as illegal tokens.

### v6: 3.38 - 3.42 (10 versions) — JSON pointer operators

Added `->` and `->>` operator tokenization in the `CC_MINUS` case, returning `TK_PTR`. The `->` operator extracts a JSON value, `->>` extracts as text. Major SQL syntax addition.

### v7: 3.43 - 3.45 (6 versions) — minor optimizations

Two small changes: (1) added `testcase(z[0]=='.')` for test coverage of dot-starting floats, (2) optimized `CC_KYWD0` to bail out early (`if(aiClass[z[1]]>CC_KYWD){ i=1; break; }`) when the second character isn't a keyword character, avoiding the keyword scan loop for single-letter identifiers.

### v8: 3.46 - 3.48 (5 versions) — digit separators

Rewrote the entire number parsing section to support `SQLITE_DIGIT_SEPARATOR` (underscore in numeric literals like `1_000_000`). Added `TK_QNUMBER` token type for numbers containing separators. The integer, hex, float, and exponent scanning loops all gained separator detection. Biggest single diff in the tokenizer's history.

### v9: 3.49 - 3.50 (4 versions) — TK_COMMENT

Changed `TK_SPACE` to `TK_COMMENT` for both `--` line comments and `/* */` block comments. Comments now get their own token type instead of being lumped with whitespace. Enables tooling (formatters, linters, syntax highlighters) to distinguish comments from spaces.

### v10: 3.51+ (2 versions) — i64 return type

Return type changed from `int` to `i64`; local variable `i` changed from `int` to `i64`. Supports tokens in SQL strings longer than 2GB. Signature-level change.

## `ai_class` — 6 variants

The `aiClass[256]` array maps each byte value to its character class constant.

### v1: 3.12 - 3.15 (4 versions) — baseline

Original character class table.

### v2: 3.16 - 3.24 (14 versions) — EBCDIC fixes

EBCDIC table corrections: `.` (0x4B) reclassified from CC_ILLEGAL to CC_DOT, `?` (0x6F) reclassified, row label typo `/* 9x */` fixed to `/* Ax */`, tilde position corrected. No ASCII changes. Most users never encounter EBCDIC.

### v3: 3.25 - 3.34 (16 versions) — CC_NUL class

Byte 0x00 changed from CC_ILLEGAL(27) to CC_NUL(28). NUL bytes get their own character class instead of being lumped with illegal characters. Matches the new `case CC_NUL:` in get_token.

### v4: 3.35.0 (1 version) — CC_KYWD0/CC_KYWD split

Major renumbering for the keyword optimization. Letters that can start an SQL keyword (A-W, a-w) get CC*KYWD0(1); letters that cannot start any keyword (X, Y, Z and their lowercase equivalents, plus `*`) get CC_KYWD(2). High bytes (0x80-0xFF) changed from CC_ID(2) to CC_ID(still 2 at this point). CC_ID, CC_ILLEGAL, CC_NUL all renumbered upward.

### v5: 3.35.5 (1 version) — EBCDIC NEL fix

EBCDIC byte 0x15 changed from CC_ILLEGAL(28) to CC_SPACE(7). 0x15 is the EBCDIC Next Line (NEL) character and should be treated as whitespace. Patch release bug fix.

### v6: 3.36+ (30 versions) — CC_BOM + high byte reclassification

Two changes: (1) high bytes 0x80-0xFE reclassified from CC_ID(2) to CC_ID(27) so the `aiClass[z[i]]<=CC_KYWD` keyword scanning loop terminates correctly with the new numbering, (2) byte 0xEF (first byte of UTF-8 BOM) assigned CC_BOM(30) for BOM detection.

## `cc_defines` — 4 variants

The `CC_*` character class constants. These track the same logical changes as `ai_class`.

### v1: 3.12 - 3.24 (18 versions) — baseline

28 character class constants (CC_X through CC_ILLEGAL).

### v2: 3.25 - 3.34 (16 versions) — +CC_NUL

Added `#define CC_NUL 28` for NUL byte handling.

### v3: 3.35 (2 versions) — +CC_KYWD0, renumbered

Added `#define CC_KYWD0 1` (first letter of keyword). CC_KYWD renumbered from 1 to 2. CC_ID renumbered from 2 to 27. CC_ILLEGAL from 27 to 28. CC_NUL from 28 to 29. Total: 30 constants.

### v4: 3.36+ (30 versions) — +CC_BOM

Added `#define CC_BOM 30` for UTF-8 BOM first byte. Total: 31 constants.

## `ctype_map` — 2 variants

The `sqlite3CtypeMap[256]` byte classification table.

### v1: 3.12 (1 version) — baseline

Original character type flags.

### v2: 3.13+ (65 versions) — quote character flag

Bytes `"` (0x22), `'` (0x27), `[` (0x5B), and `` ` `` (0x60) gained the 0x80 bit. These are the four SQL quoting characters (`"identifier"`, `'string'`, `` `identifier` ``, `[identifier]`). The new bit enables a fast `sqlite3IsQuote()` inline test used in query optimization paths.

## `upper_to_lower` — 2 variants

The `sqlite3UpperToLower[256]` case conversion table.

### v1: 3.12 - 3.35 (36 versions) — baseline

Standard 256-byte upper-to-lower mapping.

### v2: 3.36+ (30 versions) — UBSAN comparison tables appended

Added trailing comma after last ASCII entry, then appended 18 unrelated integers used for comparison operator dispatch (aLTb, aEQb, aGTb lookup tables). SQLite uses `sqlite3UpperToLower[256-OP_Ne+i]` as an indexing trick for the comparison opcodes (NE, EQ, GT, LE, LT, GE). Without these extra elements, the array access at index 256+ is technically out-of-bounds undefined behavior. The actual upper-to-lower conversion data is unchanged.

### Nature of rule removals

Every rule removal is a refactoring, never a language narrowing:

- **Alternation compression**: `BEFORE` + `AFTER` → `BEFORE|AFTER` (3.20)
- **Nonterminal renaming**: `on_opt`/`using_opt` → `on_using` (3.39), `table_options` → `table_option_set` (3.37)
- **Production restructuring**: `fullname ::= nm dbnm` → `fullname ::= nm` + `fullname ::= nm DOT nm` (3.23)
- **Feature expansion**: old rules replaced with versions that add optional suffixes (e.g., `upsert`, `returning`, `scanpt`)
- **Token class introduction**: `id` + `JOIN_KW` → `idj` (3.42)

### Conclusion

The grammar is **semantically additive** across all analyzed versions. This strongly supports using the latest grammar for parsing and performing post-parse AST validation against the target version, rather than maintaining per-version grammar variants.
